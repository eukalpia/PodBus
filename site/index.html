Overview

One API. Explicit behavior.

PodBus handles the common mechanics of message-driven services while keeping transport-specific guarantees visible in code and documentation.

Events and request/reply

Publish typed messages, consume through queue groups, or use request/reply where the selected transport supports it.

Durable workers

Run bounded workers with retry policies, dead-letter handling, durable names, and graceful drain behavior.

Database consistency

Use PostgreSQL outbox, inbox leases, and persistent idempotency when broker delivery and business state must stay aligned.

Operational visibility

Propagate W3C trace context, emit bounded Prometheus metrics, write redacted JSON logs, and expose readiness and liveness state.

Design

The abstraction has a boundary.

A message bus can normalize APIs. It cannot make NATS Core persistent, turn RabbitMQ into an append-only log, or give Kafka request/reply semantics. PodBus exposes a capability set so unsupported assumptions fail during startup.

The runtime source of truth is bus.capabilities or queue.capabilities.

Reliability

No exactly-once theatre.

Broker-backed workers use at-least-once delivery. Duplicates can occur during crashes, reconnects, lease expiry, and consumer restarts. PodBus gives you the tools to handle that reality instead of hiding it behind a slogan.

Read the reliability model
01

Acknowledge after success

The source message is acknowledged or committed only after the handler returns.

02

Make side effects idempotent

Use application keys, a shared inbox, or persistent idempotency across replicas.

03

Publish through an outbox

Store the business mutation and outgoing event in one PostgreSQL transaction.

04

Treat schemas as data contracts

Version message types explicitly and reject unknown future versions.
